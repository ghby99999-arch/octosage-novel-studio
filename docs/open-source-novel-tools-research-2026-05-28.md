# 开源小说工具调研与章鱼改造方案（2026-05-28）

## 结论

章鱼现在最该吸收的不是“更多按钮”，而是成熟写作工具的三层结构：

1. 作品导航层：书架、项目树、章节树，只放定位和状态。
2. 写作工作层：中间只服务当前正文/当前规划内容，写作时保持焦点。
3. 质量闭环层：门禁、记忆、伏笔、AI味、逻辑一致性在后台自动跑，UI 只显示可发布/需返修/返修原因。

## 代码吸收原则

章鱼当前按非盈利工具处理，因此 GPL / AGPL 项目也可以进入代码吸收候选。但要注意：GPL / AGPL 约束的核心不是“是否盈利”，而是“是否分发修改版、是否提供网络交互服务、是否保留源码和许可证”。如果未来把应用打包发给别人，或把 AGPL 代码改成在线服务，就必须同步提供对应源码、保留版权声明和许可证。

分级策略：

- MIT、Apache-2.0、BSD、ISC：可以直接引入/改造，保留 copyright 和 license notice。
- GPL、AGPL：可以吸收代码，但必须单独记录来源、保留原许可证、标记修改处；打包分发时准备公开对应源码。
- 没有明确 LICENSE 的仓库：默认不搬代码，只看交互。
- 写作规则/提示词/示例正文：不整段复制，优先提炼方法论，改成章鱼自己的网文商业规则，避免内容风格和版权风险。

落地要求：

- 新建 `docs/third-party-code-ledger.md` 记录：项目名、仓库、许可证、搬了哪些文件/思路、修改点。
- 每次引入 GPL/AGPL 代码时，在相关文件头或旁边保留来源注释。
- 如果某段代码会深度影响主应用架构，优先做成独立模块，方便后续替换或开源对应部分。
- 如果未来变成商业发行，再重新做一次许可证审计。

当前可直接看代码的优先队列：

1. Kindling：MIT。适合吸收“outline-aware editor、references panel、rolling outline、导入导出结构”。
2. LoneWriter：MIT。适合吸收“Local RAG/记忆检索、隐私优先 PWA、离线工作流”。
3. OpenWriter：MIT。适合吸收“AI 修改 diff、接受/拒绝变更、发布前人工确认交互”。
4. AgentCrew：Apache-2.0。适合吸收“多模型/多 Agent 路由配置 UI 和任务状态展示”。

GPL / AGPL 可吸收但要合规记录：

1. novelWriter：GPLv3。可学习项目树、纯文本文件结构、交叉引用和元数据组织。
2. OpenWrite：AGPLv3。可学习 Story Canvas、Codex、Analytics 概念，但不直接复制实现。
3. StoryLine：先查清许可证，再决定是否能搬代码；在此之前只学习 Corkboard、Plot Grid、Validator 的产品结构。

### Kindling

- 价值：outline-aware drafting，把大纲/场景节拍放在写作空间旁边，而不是让用户来回切。
- 可吸收：
  - 章节树点击某一章时，中间直接显示该章正文或该章细纲。
  - 章卡不是独立页面，而是正文写作时的“任务上下文”。
  - 右侧只自动露出本章相关人物、地点、物件，不堆全局信息。
- 注意：MIT，可学习 UI/结构。

来源：https://kindlingwriter.com/

### Plot Bunni

- 价值：结构清晰，acts / chapters / scenes 层级明确，并且 AI 输出支持 live streaming。
- 可吸收：
  - 开书先生成“幕/卷/章/场景”的可视化规划。
  - 网文项目树从“作品 -> 卷/阶段 -> 章节”组织。
  - 主题、字号、颜色做成真正的外观预设，而不是散落的表单。

来源：https://plotbunni.com/

### StoryLine

- 价值：信息架构最完整。它有 Corkboard、Board、Plot Grid、Timeline、Manuscript、Codex、Stats、Validator。
- 可吸收：
  - 拆书结果页做成 Plot Grid，而不是一堆 JSON 或长文本。
  - 人物关系、伏笔、时间线用图/表展示。
  - Validator 七类检查思路：时间线、人物、情节线、铺垫回收、结构、连续性/节奏、时间缺口。
  - “设定/人物/地点/伏笔”自动链接到章节。

来源：https://storyline.pixero.com/

### novelWriter

- 价值：纯文本、多文件项目树、可长期维护，不被数据库锁死。
- 可吸收：
  - 每本书必须是独立文件夹。
  - 正文、章卡、细纲、记忆、审稿报告都落文件，但 UI 用项目树统一管理。
  - 用户数据可导出、可手工查看、可迁移。

来源：https://novelwriter.io/_downloads/4273fd730e70c94b271a1e0d05c73d15/novelWriter-2.7.pdf

### OpenWrite

- 价值：AI 写作平台化，包含多 AI、Story Canvas、Codex、Analytics。
- 可吸收：
  - 模型路由在 UI 不显示模型名，只显示“写作师/审查员/修稿师/规划师”。
  - 工作流和插件化可以学习，但不要照搬。
- 风险：AGPLv3，不适合直接复制代码到商业应用。

来源：https://ilrein.github.io/openwrite/

### LoneWriter

- 价值：Local RAG + Memory & Plot Control，强调长篇一致性。
- 可吸收：
  - 写每章前从本书记忆库检索“相关前文、人物状态、金手指规则、商战资源、伏笔债”。
  - 用户可强制锁定事实，审稿和写作都必须遵守。

来源：https://alternativeto.net/software/lonewriter/about/

### StoryWriter / GOAT / DOME

- 价值：长篇生成的工程框架，而不是普通写作 UI。
- 可吸收：
  - StoryWriter：Outline Agent -> Planning Agent -> Writing Agent，先事件大纲，再章级计划，再动态压缩历史写正文。
  - GOAT：从书籍规格增强、章节大纲、章节增强、拆场景到正文，支持用户在任意尺度控制。
  - DOME：动态层级大纲 + 记忆增强，写作时规划和正文互相校正。
- 对章鱼的意义：
  - 开书不能直接给模板，必须生成并审查：项目圣经、人物关系、金手指规则、阶段大纲、前30章细纲。
  - 每章生成前必须从细纲、章卡、记忆、前文压缩上下文组装“写作任务包”。
  - 每章完成后必须更新记忆和伏笔账，再决定能否进入下一章。

来源：
- https://github.com/THU-KEG/StoryWriter
- https://github.com/GOAT-AI-lab/GOAT-Storytelling-Agent
- https://arxiv.org/abs/2412.13575

## 章鱼下一版 UI 信息架构

### 左侧主菜单

- 首页
- 创作
  - 网文创作
  - 漫剧创作
  - 拆书工坊
- 书库
- 系统配置
- 账号

删除独立的“规划中心 / 质量中心 / 投稿发布”。这些是工作流状态，不该成为主菜单。

### 网文工作台

- 左侧：项目树
  - 本书文件夹
  - 开书规划
    - 项目圣经
    - 人物关系
    - 金手指规则
    - 阶段大纲
    - 前30章细纲
  - 正文章节
    - 第001章
    - 第002章
  - 记忆账本
    - 人物状态
    - 资金账
    - 伏笔账
    - 设定锁
- 中间：当前文件/正文工作台
  - 写作时逐 token 输出
  - 审查问题直接在正文标红
  - 自动修稿时显示删除/替换动画或状态行
- 右侧：一排状态灯 + 最少量详情
  - 逻辑
  - AI味
  - 节奏
  - 爽点
  - 伏笔
  - 可发布

### 开新书页面

做成紧凑向导，不做大表单。

1. 创意卡
   - 一句话创意
   - 类型
   - 标签
   - 平台
   - 目标字数
   - 金手指
   - 主角/配角名（可空）
   - AI一键生成高质量创意
2. 书名卡
   - 调 API 生成 3-5 个书名
   - 显示标题风格原因
3. 开书规划卡
   - 生成项目圣经、人物关系、金手指规则、阶段大纲、前30章细纲
   - 自动审查逻辑，不合格自动返工
4. 进入工作台
   - 第一步不是写正文，而是展示已通过的规划树

### 拆书工坊

- 输入网址 -> 一键拆书
- 导入电子书 -> 本地拆书
- 拆书过程可视化
- 拆书结果：
  - 题材定位
  - 黄金三章结构
  - 爽点节奏
  - 人物关系
  - 伏笔回收
  - 商业化可仿写点
  - 一键仿写到新书创意

## 章鱼写作规则升级方向

1. 先修门禁误判，再修模型提示词。
2. 每个类型有独立规则包：都市重生、玄幻、系统、历史、悬疑、种田、末世等。
3. 每章只在前一章通过可发布后继续。
4. 审查不是右侧长报告，而是中间正文标红 + 右侧状态灯。
5. 未达发布线时，优先定点修补；整章重写只作为最后手段。
6. 写作必须基于“项目圣经 + 本章细纲 + 章卡 + 记忆账本 + 前文压缩”，不允许只看当前章卡硬写。

## 下一步落地顺序

1. 修第2章真实失败原因：三单/两单章卡偏离、账本证据不足、AI味和弃读段。
2. 继续跑第2章到可发布，不通过不写第3章。
3. 改项目树：把规划、人物关系、章卡、记忆都放入左侧树。
4. 改右侧状态灯：只保留一排红绿灯，详情点灯展开。
5. 重做书籍卡和开书向导，减少展示信息，换成精致组件。
6. 做拆书工坊第一版：网址/导入 -> 过程 -> 结果 -> 一键仿写。
